[INVARIANT] 2026-04-27 — Skill availability gates the prompt
Tools whose underlying permission is denied are NOT exposed in
the tool catalogue Gemma sees. Filtering happens in
RAGPipeline.buildToolSchemasAvailable(). Belt-and-braces
re-check happens in SkillRegistry.dispatch.

[INVARIANT] 2026-04-27 — Untrusted context lives in user turn
The system message contains only authoritative instructions
(identity, behavior rules, tool catalogue). Retrieved memory
and KB content always lives inside the user turn wrapped in
<untrusted> tags. PromptTemplates.sanitizeUntrustedContext()
runs before injection to strip role-spoofing tokens.

[INVARIANT] 2026-04-27 — TLS hostname allowlist
SecureHTTPSSession.allowedHosts is the ONLY set of hostnames
Eidos will TLS-handshake with. Adding a new domain requires
adding it explicitly to the set. EgressGuard backstops at a
higher level; the TLS allowlist is the second line.

[RESEARCH] 2026-04-27 — Retrieval architecture survey
Full report: research_retrieval_2026-04-27.md.
Surveyed GraphRAG (Microsoft), LightRAG (HKUST), PageIndex
(Vectify), long-context CAG, Karpathy's April 2026 LLM Wiki,
embedding model alternatives (EmbeddingGemma 308M, bge-small,
nomic-embed, gte-small, static-retrieval-mrl-en-v1), and
entity-graph patterns. Findings drove the DECISIONs below.

[DECISION] 2026-04-27 — Retrieval architecture: hybrid BM25+dense+RRF
Owner: User + Engineering. Rationale: GraphRAG / LightRAG /
PageIndex all infeasible at personal-scale on iPhone (LLM
indexing cost dominates). Long-context dump fails past 16-32K
tokens on iPhone 17 Pro Max. Hybrid retrieval (BM25 + dense
embeddings + Reciprocal Rank Fusion k=60) is 2026 best practice
for this exact use case, sub-millisecond on iOS, and SQLite
FTS5 is built into the OS. See research_retrieval_2026-04-27.md
section "Hybrid retrieval verdict".

[DECISION] 2026-04-27 — Storage: keep markdown-first
Owner: User + Engineering. Rationale: Karpathy's April 2026
"LLM Wiki" pattern (raw/ + wiki/ + ingest/query/lint operation
triad) directly validates Eidos's existing markdown-first design.
Source-of-truth stays as `Memory/<tier>/<id>.md` files. Indexes
(FTS5, vector store, entity_mentions) are derived artifacts
rebuilt from .md on change. No proprietary database. No cloud.

[DECISION] 2026-04-27 — Embedding model: NLContextualEmbedding default,
                       EmbeddingGemma 308M opt-in pilot.
Owner: Engineering. Rationale: NLContextualEmbedding is free,
ANE-accelerated, zero install. Apple doesn't publish MTEB so
quality vs alternatives is unmeasured. EmbeddingGemma 308M is
MTEB #1 sub-500M with Matryoshka truncation (768->256 dim) and
~300 MB disk; promising but unjustified without an in-house
quality eval. Pilot behind a feature flag, side-by-side, before
committing. Not blocking.

[DECISION] 2026-04-27 — Entities: tags + lazy SQLite, no graph DB
Owner: Engineering. Rationale: a real knowledge graph (nodes +
edges) breaks even only at >10K entities AND multi-hop
relational queries. Personal memory store has neither. Adopt
the LightRAG entity layer WITHOUT the LLM-driven relationship
extraction: YAML frontmatter tags as canonical, SQLite
`entity_mentions` table as derived index, no edges-as-objects.

[INVARIANT] 2026-04-27 — Storage layer ordering
Markdown files in Memory/<tier>/<id>.md are SOURCE OF TRUTH.
SwiftData rows, FTS5 tables, vector stores, entity indexes are
all DERIVED ARTIFACTS rebuilt from the markdown on change. If
markdown and any index disagree, markdown wins. Future code
must respect this — don't write to derived indexes without
mirroring to markdown first.

[INVARIANT] 2026-04-27 — Retrieval cost cap on iPhone
No retrieval path may execute more than 1 LLM forward pass per
chat turn. This rules out GraphRAG / LightRAG / PageIndex's
multi-pass query patterns and forces us toward
embedding-then-fuse approaches. The single allowed pass is the
chat reply itself (Gemma generating tokens for the user).

SECTION 2: ARCHITECTURAL DECISIONS
================================================================

[DECISION] 2026-04-27 — Stay on MLX, keep Gemma 4 as primary model
Owner: User. Rationale: Eidos's product scope (private second
brain with full phone control) demands the strongest available
multimodal model that fits the iPhone envelope; Gemma 4 E2B is
the best fit despite mlx-swift-lm's gemma4 PRs being unmerged
(Issues #389, PRs #180/#185/#187). Apple-native MLX runtime is
better for Apple Silicon than LiteRT-LM despite LiteRT-LM having
official Gemma 4 benchmarks. Net: accept community-fork risk on
mlx-swift-lm gemma4 in exchange for the model + framework fit.

[DECISION] 2026-04-26 — Disable RAG/tools/ambient on iPhone chat
Owner: Engineering. Rationale: full RAGPipeline.chat builds
10-15 K token prompts that destabilize Gemma 4 E2B on iPhone
Metal. Minimal-prompt mode is the iPhone Release default until
the safety-net cache discipline is proven stable across many
chat turns.

[RESEARCH] 2026-04-26 — Google AI Edge / LiteRT-LM evaluation
Cross-checked Google's official Gemma 4 deployment path. Found:
Google ships Gemma 4 E2B on iOS via LiteRT-LM with published
iPhone 17 Pro benchmarks (56.5 t/s decode, 0.3s TTFT, 1.45 GB
resident). Reference app: github.com/google-ai-edge/gallery.
Apple Foundation Models (iOS 26): text-only, ~3B, no third-party
models. Verdict: LiteRT-LM is the documented path for Gemma 4
on iOS today; mlx-swift-lm is community-supported. See
DECISION 2026-04-27 for which path we chose.

SECTION 3: ARCHITECTURAL INVARIANTS
================================================================

[INVARIANT] DeviceProfile.warmUp() runs on MainActor before any
background actor reads `formFactor`. Set in v6.

[INVARIANT] MLX.Memory.clearCache() between every generation is
mandatory on iPhone. Set in v9. Matches mlx-swift-lm
BenchmarkHelpers practice. Do not delete.

[INVARIANT] Every TCC permission callback / audio-engine
callback / speech-recognition callback inside a `@MainActor`
class must be `@Sendable`. Set in v6 + v10.

[INVARIANT] AVAudioSession `.record` category pairs with
`.measurement` mode. `.spokenAudio` is for playback only on
iOS 26.3.1 and returns paramErr. Set in v11.

[INVARIANT] ChatViewModel must throttle messages[i].content
updates to ~60ms intervals during streaming. Set in v12.

[INVARIANT] GemmaSession.generateText and generateMultimodal
both funnel through runGuardedGeneration(...). Lock + cache
+ thermal + cleanup are defined ONCE. New inference paths added
in the future must do the same. Set in v11.

[INVARIANT] Pre-flight memory check before any prefill. If
DeviceProfile.isMemoryConstrained, abort cleanly with
GemmaError.memoryConstrained instead of letting jetsam
reap the process. Set in v12.

[INVARIANT] Diagnostics > Smoke test pane is the regression
baseline. If Smoke fails, Gemma itself is broken. If Smoke
passes but chat fails, the wrapper layer is broken. Set in v8.

[INVARIANT] chatLite system prompt is ASCII-only, no markdown
headers, no smart quotes, no em-dashes, no instruction-content
strings like "Reminder set". Markdown content destabilizes
Gemma 4 E2B chat-template tokenization on iPhone. Set in v12.
